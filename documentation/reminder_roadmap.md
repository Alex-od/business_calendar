# План: будильники и напоминания (Radiant Days)

Документ фиксирует целевое поведение и порядок работ. Обновлять по мере реализации фаз.

**Статус:** реализовано (фазы 0–4; фаза 5 «повторы» — отложена).

---

## Продуктовая цель

**Заметка** — текст + день в календаре (`startTimeMillis`).  
**Напоминание** — отдельное время push (`alarmTimeMillis`), не путается с днём заметки.  
**Выполнено** — заметка закрыта для напоминаний, в списке только приглушается (не скрывается).

---

## Зафиксированные решения (2026-06-03)

| # | Вопрос | Решение |
|---|--------|---------|
| 1 | Время будильника по умолчанию при добавлении | **Сейчас + 1 час** (не `startTimeMillis` заметки) |
| 2 | Выполненные заметки в списке дня | **Только приглушать** (зачёркивание / пониженная контрастность), не скрывать |
| 3 | Повторы (ежедневно, по будням) | **Позже** — отдельная фаза, не в ближайшем релизе |
| 4 | Точность напоминаний | **WorkManager достаточно**, допустимое отклонение **~5 минут** (без exact alarms / `SCHEDULE_EXACT_ALARM`) |
| 5 | `notificationMinutesBefore` в модели | **За N минут до `alarmTimeMillis`** (не до `startTimeMillis`). Push в `(alarmTimeMillis - N)`. Если будильника нет — поле не используется. |

---

## Текущее состояние (baseline)

- `alarmTimeMillis` — опционально; push через `EventNotificationWorker` (опрос **каждые 15 мин**).
- Добавить / удалить будильник в тулбаре (`AlarmAdd` / `AlarmOff`).
- `isCompleted` сохраняется в БД, **не влияет** на UI календаря и уведомления.
- `POST_NOTIFICATIONS` в манифесте; **нет** runtime-запроса на Android 13+.
- Деструктивные миграции Room **отключены** (`CalendarDatabaseMigrations`, без `fallbackToDestructiveMigration`).

---

## Целевой user flow

1. Создаю заметку → при необходимости включаю напоминание (по умолчанию **+1 ч**).
2. В сетке месяца / списке дня вижу иконку будильника.
3. В назначенное время (±5 мин) приходит **один** push.
4. Из push: открыть заметку; позже — «Выполнено» / «Отложить».
5. «Выполнено» → больше не напоминать; в списке строка **приглушена**.

## Фазы реализации

### Фаза 0 — Модель данных (½ дня)

- [x] `notificationMinutesBefore` = минуты **до `alarmTimeMillis`** (решение зафиксировано).
- [x] Планировщик: push в `alarmTimeMillis - notificationMinutesBefore`.
- [x] UI: 0 / 5 / 15 / 30 мин при включённом будильнике; по умолчанию 0.
- [x] `onAddAlarmClick`: **сейчас + 1 ч**.
- [x] KDoc в `CalendarEvent` + блок «Напоминание» в форме.

### Фаза 1 — UX «понятно и быстро» (2–3 дня)

- [x] Блок «Напоминание» с временем push в форме.
- [x] Пресеты: +15 мин, +1 ч, завтра 9:00, день заметки 9:00.
- [x] Иконка будильника в сетке и списке дня.
- [x] Выполнено → приглушение и зачёркивание.
- [x] `POST_NOTIFICATIONS` при добавлении будильника; текст в Settings.

**Критерий:** за ~10 с понятно, когда придёт push.

### Фаза 2 — Надёжные уведомления (3–5 дней)

- [x] `AlarmScheduler` + `ReminderNotificationWorker` (one-time, unique per note).
- [x] Перепланирование при save/update; cancel при удалении будильника/заметки.
- [x] Не планировать без будильника или при `isCompleted`.
- [x] `ReminderNotifierStore` + unique work против дублей.
- [x] `ReminderBootReceiver` на `BOOT_COMPLETED`.
- [x] `ReminderFallbackWorker` раз в 12 ч (вместо опроса каждые 15 мин).

**Критерий:** напоминание в целевое время ±5 мин, без повторов.

### Фаза 3 — Действия из уведомления (2–3 дня)

- [x] Tap → редактирование заметки по `eventId`.
- [x] «Выполнено» / «+5 мин» / «+10 мин» в уведомлении.

### Фаза 4 — Виджет и фильтры (1–2 дня)

- [x] Виджет: только напоминания в окне 24 ч (SQL `is_completed = 0`).
- [x] Bridge без изменений контракта.

### Фаза 5 — Повторы (отложено)

- [ ] Ежедневно / по будням / еженедельно.
- [ ] Отдельная таблица или поля `repeatRule` — проектировать при старте фазы.

**Не делать до явного запроса:** exact alarms, несколько будильников на одну заметку, backend для push.

## Архитектура (минимальная)

| Слой | Ответственность |
|------|------------------|
| `:domain` | Use-case: `ScheduleReminder`, `CancelReminder`; правила «когда не слать» |
| `:data` | Room без лишних полей; миграции только явные |
| `:app` | `AlarmScheduler`, workers, UI пресеты, permission flow |

Не расширять WebSocket/MCP ради напоминаний, пока локальный сценарий не стабилен.
